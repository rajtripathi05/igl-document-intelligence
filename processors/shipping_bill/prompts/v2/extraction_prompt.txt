Analyze the attached Export Shipping Bill (Indian Customs / ICEGATE).

Convert it into the standardized customs/ERP JSON schema.

The document is usually an ICEGATE "LET EXPORT COPY" / "SHIPPING BILL SUMMARY".

═══════════════════════════════════════════════════════
DOCUMENT UNDERSTANDING
═══════════════════════════════════════════════════════

Shipping Bills are NOT unstructured documents.

They are standardized Government of India Customs documents generated through
the ICEGATE / CBIC system. The overall layout, section ordering, field labels,
and business structure remain largely consistent across Shipping Bills.

Therefore, extraction must NOT rely on semantic guessing.

First understand the document structure, then extract each value from the
appropriate section. Treat the Shipping Bill as a structured government form,
not a generic PDF. The uploaded sample Shipping Bills represent the standard
government format and are the reference layout.

═══════════════════════════════════════════════════════
STRUCTURAL PARSING (do this BEFORE any extraction)
═══════════════════════════════════════════════════════

Before extracting any business field, perform a structural analysis of the
whole document and build an internal map of its sections. Typical sections:

• Shipping Bill Header      • Exporter Details      • Buyer / Consignee Details
• Invoice Details           • Item Details          • Port Details
• Drawback Details          • RODTEP / Export Promotion Details
• Process Details           • Examination Details   • Assessment Details
• LEO Details               • Declaration           • Summary

Only after the complete structure has been identified should business
extraction begin. Do NOT extract values while scanning pages. First understand.
Then extract.

If multiple pages exist, MERGE them into one logical Shipping Bill before
extraction. Never process pages independently. Header/invoice details, item
tables, weights, drawback/RODTEP amounts, commission, exchange rate and
LEO/examination details are frequently spread across several pages.

═══════════════════════════════════════════════════════
SECTION-BASED EXTRACTION
═══════════════════════════════════════════════════════

Every business field belongs to a predefined section of the Shipping Bill.
Always navigate directly to the expected section before extraction. NEVER
search the entire document for the first matching number. Use the hierarchy
below (field → owning section → JSON target).

── Shipping Bill Header ──────────────────────────────
• Port Code            → shipping_bill.port_code
• Shipping Bill Number → shipping_bill.shipping_bill_number
• Shipping Bill Date   → shipping_bill.shipping_bill_date
  Port Code, SB No. and SB Date appear together in the top header band
  (e.g. "INDEL4", "4525369", "27-JUN-26"). Capture precisely; do NOT confuse
  the Job/CB code with the SB number.
• Customs House        → shipping_bill.customs_house
  The named office in the header (e.g. "NEW CUSTOM HOUSE, IGI AIRPORT, NEW DELHI").
• Export Promotion Scheme → shipping_bill.export_promotion_scheme
  RODTEP, EPCG, DBK or other scheme codes in the header/scheme block.

── Port Details ──────────────────────────────────────
• Port of Loading        → shipping_bill.port_of_loading
• Port of Discharge      → shipping_bill.port_of_discharge  (foreign port of unloading)
• Country of Discharge   → shipping_bill.country_of_destination
  These appear in the numbered summary block (items 12–15).

── Exporter Section ──────────────────────────────────
• Exporter name / address / IEC (Branch) number / GSTIN
  → exporter.name, exporter.address, exporter.iec_number, exporter.gstin

── Buyer / Consignee Section ─────────────────────────
• Foreign party — name/address may be printed as several "By Name" lines.
  Combine name into consignee.name; combine address lines into consignee.address.
  → consignee.name, consignee.address, consignee.country
• Set the buyer block from the destination information.
  → buyer.name, buyer.country

── Invoice Details ───────────────────────────────────
• Invoice Number   → invoice.invoice_number
• Invoice Date     → invoice.invoice_date
• FOB VALUE (FC)   → invoice.fob_value      (see FIELD DISAMBIGUATION — use the
                                             FOREIGN CURRENCY FOB, never FOB in INR)
• Invoice Value    → invoice.invoice_value  (total FC invoice value)
• Currency         → invoice.currency
• Freight          → invoice.freight        (from the INVOICE section only)
• Insurance        → invoice.insurance      (from the INVOICE section only)
• Commission       → invoice.commission
• Exchange Rate    → invoice.exchange_rate   (customs-notified INR per FC unit)

  Commission (CRITICAL — NEVER OMIT): labels "COMMISSION", "6. COMMISSION",
  "AGENT'S COMMISSION", "BUYING/SELLING COMMISSION"; often item 6 in the
  invoice value breakdown. If printed 0 / 0.00 / NIL / ZERO, extract 0 — never
  null. Return null only if no commission field exists anywhere.

  Exchange Rate labels: "EXCHANGE RATE", "EX RATE", "RATE OF EXCHANGE",
  "CUSTOMS EXCHANGE RATE", "CURRENCY RATE" (e.g. 83.50, 93.60, 84.10).

── Item Details ──────────────────────────────────────
• HS Code   → items[].hs_code
  Labels: "HS CODE", "HS CD", "HSN", "HS No.", "TARIFF CODE", "CTH",
  "CUSTOMS TARIFF HEADING", "CHAPTER HEADING". Extract exactly as printed
  (8-digit code, e.g. "29094100"). Do not normalize or truncate.
• Quantity  → items[].quantity
• UOM (UQC) → items[].unit
• Rate      → items[].unit_price

── Process / LEO Details ─────────────────────────────
• LEO Date  → shipping_bill.leo_date
  Labels: "LEO Date", "Let Export Order Date", "LEO DT", "LEO",
  "LET EXPORT DATE", "L.E.O. DATE". Date the customs officer issued the Let
  Export Order — often one day after the SB date. Normalize to YYYY-MM-DD.
  If handwritten and legible, prefer the handwritten value.
• LEO Number → shipping_bill.leo_number

── Export Promotion (RODTEP) Details ─────────────────
• RODTEP Amount → drawback.rodtep_amount
  Labels: "RODTEP AMOUNT", "RODTEP AMT", "RoDTEP AMOUNT", "REMISSION AMOUNT",
  "RODTEP INCENTIVE AMOUNT". This is the actual RODTEP monetary amount in INR.
  If 0 / 0.00, extract 0 — never null.

── Drawback Details / Drawback Summary ───────────────
• DBK Rate (%)  → drawback.dbk_rate
  Labels: "DRAWBACK RATE", "DBK RATE", "DBK %", "RATE OF DRAWBACK".
  In the ICEGATE "A. DRAWBACK & ROSL CLAIM" table the drawback row is laid out
  as: INV SNO | ITEM SNO | DBK SNO | QTY/WT | VALUE | RATE | DBK AMT | ...
  The DBK Rate is the "RATE" column of THAT drawback row (e.g. 1.2), sitting
  immediately before the DBK AMT column. Extract only the numeric value
  (e.g. 1.10 for "1.10%", 1.2 for a "1.2" rate).
  Do NOT take the Exchange Rate, the item "6.RATE"/unit rate, or the CESS RATE.
  Relationship (for understanding only — do NOT compute it yourself; Python
  derives it when the printed rate is absent):
  DBK Rate (%) = DBK Amount (INR) ÷ FOB Value in INR × 100, where
  FOB Value in INR = FOB (Foreign Currency) × Customs Exchange Rate.
  If the RATE column is genuinely not printed, return null for dbk_rate.
• DBK Amount    → drawback.dbk_amount
  Labels: "DRAWBACK AMOUNT", "DBK AMT", "DUTY DRAWBACK AMT", "AMOUNT OF
  DRAWBACK". The computed/sanctioned drawback amount in INR. If 0, extract 0.
• DBK Claim     → drawback.drawback_amount
  The total drawback claimed / claim reference amount. If identical to DBK
  Amount in the document, use the same value for both.
• Scheme Description → drawback.scheme_description
  Labels: "SCHEME DESCRIPTION", "SCHEME DESC", "EXPORT SCHEME". Extract the
  complete text exactly as printed (e.g. "REMISSION OF DUTIES AND TAXES ON
  EXPORTED PRODUCTS", "DUTY DRAWBACK SCHEME").

═══════════════════════════════════════════════════════
FIELD DISAMBIGUATION
═══════════════════════════════════════════════════════

The Shipping Bill frequently contains multiple values with similar names.
Do NOT choose the first matching value — identify the correct business section.

• FOB Value — the document may show FOB (INR) and FOB (Foreign Currency).
  ALWAYS extract FOB (FOREIGN CURRENCY) into invoice.fob_value.
• Freight — may appear in Invoice, Summary and Calculation blocks.
  ALWAYS use the Invoice section value.
• Insurance — may appear multiple times. ALWAYS use the Invoice Details value.
• LEO Date — NEVER confuse with Assessment Date, BRC Date, Realisation Date,
  Submission Date or Examination Date.
• DBK Amount — NEVER confuse with DBK Claim, RODTEP, Duty Saved or Refund.
• RODTEP Amount — NEVER confuse with Quantity, SQC, Weight or Package Count.
• Exchange Rate — NEVER confuse with FOB Conversion, INR Value or Duty Rate.

═══════════════════════════════════════════════════════
LINE ITEM FIELDS
═══════════════════════════════════════════════════════

• Extract every product row: line number, product description, HS code,
  quantity, unit (UQC), unit price, FOB value, gross weight, net weight,
  number of packages.
• Preserve line item order.
• Preserve quantities, units, weights and prices exactly.

═══════════════════════════════════════════════════════
ZERO VALUE RULES
═══════════════════════════════════════════════════════

Never leave a numeric field null/blank simply because its value is zero.
If the document explicitly prints 0, 0.00, NIL or ZERO for any of these
fields, extract the value as 0 (numeric zero):
• invoice.commission     • invoice.freight        • invoice.insurance
• drawback.drawback_amount  • drawback.dbk_amount  • drawback.rodtep_amount

Return null ONLY when the field genuinely does not appear anywhere in the
document. Do not return null because of OCR difficulty or a page you skipped.

═══════════════════════════════════════════════════════
HANDWRITING & OCR
═══════════════════════════════════════════════════════

• Support handwritten values, corrections, abbreviations and poor OCR.
• When handwritten values override printed values, prefer the handwritten
  value when confidence is high.
• Common handwritten fields: LEO Date, LEO Number, Exchange Rate corrections.

═══════════════════════════════════════════════════════
STRUCTURAL VALIDATION (after extraction, before returning)
═══════════════════════════════════════════════════════

Verify every field against the expected document structure. The section source
is as important as the value — an incorrect section means an incorrect
extraction. Reject and re-extract if:

• FOB VALUE (FC) was taken from anywhere other than Invoice Details
  (or was the INR FOB instead of the Foreign Currency FOB).
• LEO Date was taken from anywhere other than Process / LEO Details.
• DBK Rate was taken from anywhere other than Drawback Details.
• RODTEP Amount was taken from anywhere other than Export Promotion Details.
• Freight or Insurance were taken from the Summary/Calculation block instead
  of Invoice Details.

═══════════════════════════════════════════════════════
GENERAL REQUIREMENTS
═══════════════════════════════════════════════════════

• Preserve FOB value, freight, insurance, invoice value and exchange rate exactly.
• Preserve HS codes, IEC number, GSTIN, port codes, container and seal numbers exactly.
• Preserve Shipping Bill number, Shipping Bill date, LEO number and LEO date exactly.
• Capture the shipping scheme, export promotion scheme and drawback details.
• If a field is genuinely unavailable anywhere in the document, return null.
• Never guess values. Never invent values.
• If calculations do not match, preserve document values and report the mismatch.
• Return ONLY valid JSON matching the schema below.
• Do not explain reasoning. Do not return markdown.

═══════════════════════════════════════════════════════
PRIMARY EXTRACTION STRATEGY
═══════════════════════════════════════════════════════

This is a Government of India structured document. Do NOT use a generic
document-extraction strategy — use a template-aware strategy. The document
follows a fixed government layout: navigate using section headers, field
labels, table layouts and business hierarchy. Prefer structural understanding
over semantic similarity.

Extract fields exactly as a trained customs officer would read the Shipping
Bill. Accuracy is more important than speed. Never guess when the document
structure clearly indicates the correct source section.
